02What is the difference between load, stress, spike, soak, and endurance testing?Concepts▾
Your Answer
Load: Expected peak concurrent users — validates normal operating capacity. Stress: Beyond peak — finds the breaking point and how gracefully the system fails. Spike: Sudden massive burst — simulates flash sale or news event traffic. Soak/Endurance: Sustained normal load over hours/days — catches memory leaks and connection pool exhaustion that only appear over time. Scalability: Incremental load increase — validates that performance scales proportionally with infrastructure. For Marriott I'd prioritise soak and spike tests around peak travel seasons and promotional events.

03How do you identify and diagnose a performance bottleneck in a microservices system?Diagnosis▾
Your Answer
Start at the top — JMeter or NeoLoad shows which transaction is slow. Then Dynatrace distributed traces show exactly which service and method is responsible. Then drill into that service — is it CPU, memory, I/O, or database? Dynatrace hotspots and Splunk log correlation tell me if it's a slow query, a connection pool exhaustion, or a downstream dependency. The key workflow: load generator shows symptom → APM shows location → infrastructure metrics show cause → code profiler shows root. I never guess — I follow the trace.

05What is service virtualization and when would you use it in performance testing?Concepts▾
Your Answer
Service virtualization simulates the behavior of dependent services that aren't available or can't be load-tested. Use it when: a downstream service isn't ready yet, you can't fire load at a third-party payment processor, or the real service would be overwhelmed by test traffic. Using ReadyAPI I create a virtual service that responds at a realistic latency — say 120ms average with 5% variance — so the service under test behaves as if the dependency is real. This isolates exactly what I'm testing and prevents test results being corrupted by a slow or unavailable dependency. For Marriott this is critical for payment gateways and loyalty point APIs.

08Walk me through a JMeter test plan structure for a Marriott booking API.JMeter▾
Your Answer
Thread Group defines virtual users and ramp-up — I typically ramp over 5 minutes to avoid artificial spikes. HTTP Request Defaults sets the base URL and headers so they're not repeated per sampler. Config Elements handle CSV Data Set for parameterised booking data and HTTP Cookie Manager for session handling. Samplers cover each API call in the booking flow: search availability → select room → create reservation → confirm. Assertions validate response codes (200, 201) and response body fields — not just "no error" but "reservation ID is non-null." Listeners are disabled during actual runs and replaced with backend listeners pushing to InfluxDB or Grafana for real-time dashboards. Post-processors handle dynamic correlation — extracting session tokens from login response and injecting into subsequent requests.

09What is correlation in performance testing and why does it matter?Concepts▾
Your Answer
Correlation extracts dynamic values from server responses and injects them into subsequent requests. Without it, every virtual user sends the same hardcoded session token — the server rejects all but the first, and your load test is measuring error handling, not real performance. Common values to correlate: session tokens, CSRF tokens, reservation confirmation numbers, loyalty account IDs, and OAuth bearer tokens. In JMeter I use Regular Expression Extractor or JSON Extractor post-processors to capture these values from responses and store them in variables. For Marriott's booking system with SSO and loyalty session management, correct correlation is the difference between a meaningful test and 10,000 401 errors.

11How do you approach performance tuning when a bottleneck is identified?Tuning▾
Your Answer
Performance tuning is a hypothesis-test cycle, not a shotgun approach. I identify the single biggest bottleneck first — Dynatrace method hotspots usually make this obvious. I form a hypothesis: "this slow query is causing the p95 spike" — and validate it by running the isolated query in the database with EXPLAIN ANALYZE. Then I implement one fix at a time — index addition, query rewrite, caching layer, connection pool increase — and retest. This is critical: changing multiple things simultaneously means you don't know what helped. Common fixes: slow SQL queries (index or rewrite), connection pool exhaustion (increase pool size or reduce connection hold time), thread contention (async processing), and downstream API latency (caching or circuit breaker).
Splunk + SignalFx debugging · Care.com SRE collaboration · production bottleneck prevention
12How do you test Kubernetes autoscaling behavior under load?Kubernetes▾
Your Answer
I test three autoscaling behaviors: scale-out speed, scale-in safety, and failure recovery. Scale-out: ramp load past the HPA threshold and measure how long before new pods are ready and traffic-serving — if it takes 3 minutes to scale but response times degrade in 30 seconds, the HPA threshold needs to trigger sooner. Scale-in: reduce load gradually and validate that pods terminate gracefully without dropping in-flight requests — connection draining must be configured correctly. Failure recovery: kill a pod during peak load and validate circuit breakers engage and the remaining pods handle the traffic surge without cascading failure. I monitor all of this via Dynatrace or Grafana showing pod count, CPU utilisation, and response time simultaneously.

13JMeter vs NeoLoad — when do you choose one over the other?Tools▾
Your Answer
JMeter: open-source, highly extensible via plugins, excellent for scripted HTTP/REST/GraphQL load, integrates cleanly into Jenkins and Harness pipelines, large community. I use it as my default for API-level load testing. NeoLoad: commercial, stronger GUI for non-developer testers, superior protocol support for WebSockets, complex session management, SAP, and thick-client protocols, built-in CI/CD connectors, and better built-in correlation for dynamic session handling. For Marriott's complex SSO and loyalty session flows, NeoLoad's automatic correlation is a genuine advantage over JMeter's manual extractor approach. My rule: JMeter for API and REST-heavy services, NeoLoad when the protocol complexity or team skill set makes GUI-driven scripting preferable.

15How do you use Splunk to correlate application logs with performance test events?Splunk▾
Your Answer
I timestamp the start and end of each JMeter test run and use those as Splunk time filters. During the test window I run SPL queries for error rates by service, exception stack traces, slow database query logs, and timeout patterns. The key technique is correlating JMeter's transaction timestamps with Splunk log entries — if JMeter shows a spike in booking-service latency at 14:23:45, I query Splunk for that service's logs in that 30-second window. Common findings: sudden burst of "connection pool exhausted" log entries perfectly correlating with the JMeter latency spike, or "downstream timeout" errors from the payment service appearing exactly when virtual user count crossed 500. 20How do you use APM tools to establish a performance baseline before a test?APM▾
Your Answer
A baseline is the agreed "normal" state — the performance fingerprint of the system before a test or deployment. I capture it from Dynatrace production data over a representative two-week window: p50, p90, p95, p99 response times per service, throughput (TPS), error rate, CPU and memory utilisation at average and peak. This baseline becomes the comparison point for every subsequent performance test — I'm not just checking "did we pass the SLA" but "did this release change the system's performance profile?" A deployment that stays under SLA but increases p99 by 40% is a warning signal. Without a baseline you're blind to regression. 21What is the role of chaos engineering in a performance testing program?Chaos▾
Your Answer
Chaos engineering tests resilience — what happens when things fail under load, not just when everything works. It complements performance testing by answering: does the system degrade gracefully or catastrophically when a pod dies during peak traffic? Do circuit breakers engage before a slow downstream service causes a cascade? Does the load balancer redistribute correctly when a node drops? In practice I run chaos experiments alongside load tests — kill a replica while 80% peak load is running and measure recovery time and error rate spike. Tools: Chaos Monkey, AWS Fault Injection Simulator, or simple Kubernetes pod deletion. The finding that a 3-second downstream timeout without a circuit breaker cascades to a 30-second user-facing failure is exactly what chaos engineering catches before a production incident does.

22How do you handle SSL/TLS and authentication in performance test scripts?Security▾
Your Answer
SSL: JMeter handles HTTPS natively — I disable certificate validation for test environments using the jmeter.httpclient.reset_state_on_thread_group_iteration property to avoid certificate errors blocking the test. Authentication: I never hardcode credentials. For OAuth2/JWT I script a login transaction at the start of the thread group that captures the bearer token via JSON Extractor and stores it in a user-defined variable. All subsequent requests inject it as an Authorization header. For Marriott's SSO environment with rotating tokens, I include a token refresh step that fires before expiry to prevent mid-test authentication failures. API keys are stored in JMeter property files that are injected at runtime by the CI/CD pipeline from a secrets manager — never in the test plan file itself.

23How do you embed performance gates in a CI/CD pipeline without slowing every build?CI/CD▾
Your Answer
Tiered execution: every commit triggers a lightweight smoke performance test — 50 virtual users for 2 minutes on critical endpoints only. This runs in 5 minutes and catches obvious regressions without blocking developers. Full load tests (500+ users, 30 minutes) run nightly or on release branch creation only. This way developers get fast feedback on every push and the full performance profile is validated before every release. Thresholds are different per tier — smoke gate fails if p95 exceeds 3 seconds (clear regression), full gate fails if p95 exceeds the agreed SLA. At Care.com I implemented this two-tier approach and improved CI/CD reliability by 90% while keeping pipeline time reasonable.

26How do you test Kubernetes pod autoscaling under load?Kubernetes▾
Your Answer
Three-phase test: Phase 1 — baseline at steady state, confirm pod count and resource utilisation are stable. Phase 2 — ramp load beyond HPA CPU threshold (typically 70-80%) and measure: how long before HPA triggers, how long before new pod is ready and serving, and whether response time degrades during the scale-out window. Phase 3 — reduce load and validate graceful scale-in — pods must drain existing connections before terminating, confirmed by zero 5xx errors during scale-down. I instrument all of this with Dynatrace showing pod count, CPU, and response time on the same timeline so the correlation is clear. Common finding: HPA threshold too high — pod count stays at minimum until the system is already degraded. Fix: lower the trigger threshold and add a buffer replica at all times for hospitality peak scenarios.

28What is TDD and BDD and how do you drive adoption in a resistant team?TDD/BDD▾
Your Answer
TDD: write the failing test first, then the minimal code to pass it. Benefits: forces clear interface design, gives instant regression coverage, reduces debugging time. BDD extends this to business language — Gherkin scenarios (Given/When/Then) that product, QA, and developers write together before implementation. Benefits: requirements are unambiguous before a line of code is written, and the scenarios become executable tests. Adoption strategy: pilot on one team with one service. Track rework cycles before and after. The data persuades where mandates fail. For BDD specifically I facilitate a Three Amigos session — product owner, developer, and QA — to write the scenarios together. That shared ownership is what makes BDD succeed. At Care.com I drove shift-left practices that moved QA involvement from post-development to sprint planning — same cultural challenge.

36How do you approach risk identification and monitoring in a QE program?Risk▾
Your Answer
Risk-based testing prioritises effort where failure impact is highest. I build a risk matrix at the start of each release cycle: what changed (code diff analysis), who is affected (customer-facing vs internal), what is the blast radius if it fails (single user vs all booking traffic), and what is the probability of failure (change complexity, developer experience with this module, recent defect history). High-risk changes get full regression — low-risk changes get smoke tests. I also maintain a risk register for QE: environments that are consistently unreliable, dependencies with no service virtualization, test areas with coverage gaps. This register is reviewed monthly with engineering leadership so risks are visible and owned.

42How do you establish a code review process within the QA team?Code Review▾
Your Answer
