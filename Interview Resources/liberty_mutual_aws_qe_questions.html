01How do you test AWS Lambda functions — what's unique vs testing traditional services?Serverless▾
3-sentence answer
Lambda functions have unique testing challenges: they're stateless and event-driven, so tests must inject events rather than making HTTP calls, and they have cold start latency that doesn't exist in traditional services. I test at three levels — unit tests that invoke the handler function directly with synthetic events (fast, no AWS dependency), integration tests that deploy the function to a real AWS test account and invoke it with the AWS SDK (catches IAM, VPC, and service integration issues), and end-to-end tests that trigger the full event chain from source to Lambda to downstream system. Cold start behaviour must be explicitly tested for latency-sensitive functions — I measure p99 latency including cold start, not just warm execution.
Liberty Mutual processes 100 million transactions per month through Lambda-based FCS. Cold start latency on Lambda functions handling financial transactions is a critical performance characteristic to test.
02How do you test AWS Step Functions workflows?Serverless▾
3-sentence answer
Step Functions orchestrate sequences of Lambda functions and AWS services — testing them requires validating state machine logic, error handling paths, and retry behaviour, not just the happy path execution. I use the AWS Step Functions Local emulator for unit testing state machine logic without cloud costs, and deploy to a sandbox account for integration testing against real AWS services. Critical test scenarios: successful execution through all states, failure handling at each state (does the error branch execute correctly?), retry logic (does the function retry the correct number of times with the right backoff?), and timeout behaviour (does the workflow handle Lambda timeouts gracefully?).
Liberty Mutual's FCS system uses Step Functions to orchestrate ETL processes for financial data from global business lines. Testing these workflows is directly relevant to the GRS-QE role.

05How do you test AWS EventBridge and SNS/SQS event-driven architectures?Serverless▾
3-sentence answer
Event-driven systems are harder to test because there's no synchronous request-response — I publish an event and must verify downstream effects, which requires polling or subscribing to the outcome rather than reading a return value. My approach: publish a test event to EventBridge or SNS, then poll the expected downstream state (DynamoDB record updated, SQS message consumed, Lambda execution logged in CloudWatch) until the assertion passes or a timeout triggers failure. I also test event routing rules explicitly — does the right Lambda get triggered by the right event pattern, and does an unexpected event pattern correctly fail to trigger anything — because misconfigured routing is a silent failure mode in event-driven systems.

06How do you performance test Lambda functions — what metrics matter?Serverless▾
3-sentence answer
Lambda performance has three unique dimensions traditional services don't: cold start duration (time to first invocation after a period of inactivity), warm execution duration (subsequent invocations — this is the p95/p99 you SLA against), and concurrency limits (Lambda scales automatically but has account-level and function-level concurrency limits — hitting the limit causes throttling, which looks like a timeout to the caller). I test all three: cold start via intentional function dormancy followed by a burst of invocations measured with CloudWatch Lambda Insights, warm performance via sustained load using JMeter or k6 invoking the Lambda through API Gateway, and concurrency limits by deliberately setting a low reserved concurrency and observing throttling behaviour. For Liberty's FCS processing 100 million monthly transactions, the concurrency model of the financial processing Lambda is a critical performance characteristic.
$60 per million transactions is Liberty's target cost. Lambda memory allocation directly affects both performance and cost — testing the memory-performance tradeoff is a meaningful QE contribution at Liberty.
07How do you test DynamoDB-backed serverless applications?Serverless▾
3-sentence answer
DynamoDB testing has two distinct concerns: data model correctness (does the access pattern work efficiently — no full table scans, partition key design is valid?) and throughput behaviour (does the application handle DynamoDB throttling gracefully when read/write capacity is exceeded?). For data model testing I use DynamoDB Local for unit tests — fast, free, no AWS dependency — and validate that queries return the expected items with the expected attributes for a range of partition key distributions. For throughput I deliberately set low provisioned capacity and run a load test that forces throttling — the application must implement exponential backoff and retry, and I assert that no data is lost and no errors propagate to the end user when throttling occurs.

09How do you use AWS CloudWatch Logs Insights for test failure diagnosis?AWS Testing▾
3-sentence answer
CloudWatch Logs Insights lets me query structured logs across all Lambda executions with a SQL-like syntax — for test failure diagnosis I query for error patterns, exception stack traces, and slow execution durations in the time window of the failing test run. My standard diagnostic query: filter @type = "REPORT" | stats max(@duration), min(@duration), avg(@duration), count() by bin(5m) — this shows me the Lambda execution time distribution during the test, immediately revealing if a performance regression is consistent or intermittent. I also correlate Lambda execution IDs from the test run with CloudWatch log entries — the test framework logs the request ID, and I query CloudWatch for that specific ID to see the complete execution context for a failing test.
Knowing specific CloudWatch Insights query syntax signals hands-on AWS experience — not just "I use CloudWatch."
10What is AWS X-Ray and how do you use it in QE?AWS Testing▾
3-sentence answer
AWS X-Ray provides distributed tracing for serverless and microservices applications — it maps the complete call chain across Lambda functions, API Gateway, DynamoDB, and external services, showing exactly where time is spent in each transaction. I also use X-Ray traces as test assertions — if the trace shows a DynamoDB call that wasn't expected, that's a test finding indicating the Lambda is making an unplanned external call that could affect performance under load.

11How do you test AWS API Gateway — what are the unique failure modes?AWS Testing▾
3-sentence answer
API Gateway sits in front of Lambda and adds its own failure modes independent of the Lambda function: throttling (429 Too Many Requests when per-method or account-level limits are hit), authoriser failures (Lambda authorisers that reject valid tokens or accept invalid ones), and integration timeout (API Gateway has a maximum 29-second integration timeout — a Lambda that runs longer returns a 504 regardless of whether it completes). I test all three explicitly: throttling by running burst load above the configured rate limit and asserting the 429 response includes a Retry-After header; authoriser by testing valid, expired, and malformed tokens; timeout by deploying a Lambda with a deliberate sleep and validating the 504 response. For Liberty's financial APIs, authoriser testing is especially critical — a misconfigured authoriser that rejects valid insurance agent tokens blocks business workflows.

12How do you use the AWS SDK in test automation?AWS Testing▾
3-sentence answer
The AWS SDK is the test automation's primary tool for state verification in serverless testing — rather than testing through the UI or HTTP, I use the SDK to directly assert on the state of AWS resources after a test action. Example patterns: after an order placement API call, I use the DynamoDB SDK to query the orders table and assert the record was created with correct attributes; after a file upload triggers a Lambda, I use the S3 SDK to verify the processed output file exists in the destination bucket; after a payment event, I use the SQS SDK to verify the downstream notification message was queued with the correct payload. This direct state validation is faster and more reliable than testing through the full UI stack and it validates the serverless contract — the right data reached the right AWS resource.
AWS SDK-based state verification is the key testing pattern for serverless — it's what separates candidates who've actually tested serverless from those who've tested traditional APIs.

14What is AWS SAM and how does it help with local serverless testing?AWS Testing▾
3-sentence answer
AWS SAM (Serverless Application Model) provides a local Lambda and API Gateway emulator — sam local invoke runs a Lambda function locally using Docker, and sam local start-api starts a local HTTP server that routes requests to Lambda handlers, enabling fast unit and integration testing without AWS round-trips or costs. 16How do you test AWS IAM permissions — what QE responsibility exists here?AWS Testing▾
3-sentence answer
IAM permission testing validates that the principle of least privilege is correctly implemented — the Lambda function can access exactly what it needs and nothing more. I test both directions: positive (the Lambda can perform its required actions — write to its DynamoDB table, publish to its SNS topic) and negative (the Lambda cannot perform actions outside its scope — cannot read another service's DynamoDB table, cannot write to S3 buckets it doesn't own). I automate this using the AWS IAM Policy Simulator API in the test suite — given a role ARN and a set of actions and resources, it returns Allow or Deny without making the actual API calls, so I can assert permission boundaries without incurring service costs or creating test data side effects.
The IAM Policy Simulator API is a specific, practical tool that shows real AWS QE depth. Most candidates describe IAM testing conceptually — naming the specific tool differentiates you.

21How do you prevent configuration drift in AWS environments used for testing?CI/CD & IaC▾
3-sentence answer
Configuration drift — where the actual AWS resource configuration diverges from what the CDK or Terraform code says it should be — is a silent test validity risk: tests pass against a drifted environment that doesn't represent production, giving false confidence. I prevent drift by treating test environments as immutable — never manually modifying them, always destroying and redeploying via CDK when a change is needed. I also run AWS Config rules as automated drift detection — Config continuously evaluates resource configurations against defined rules and alerts when a resource drifts from its expected state, giving me a signal that the test environment is no longer a valid testing surface before I waste a testing cycle against it.

27What is distributed tracing and how does X-Ray implement it?Observability▾
3-sentence answer
Distributed tracing tracks a single request as it flows through multiple services — each service adds a span to the trace, recording its start time, duration, and any errors, so the complete execution path is reconstructable from a single trace ID. X-Ray implements this by injecting a trace header (X-Amzn-Trace-Id) into every API Gateway request, which Lambda reads and propagates to all downstream SDK calls — DynamoDB, SNS, Step Functions — automatically, with no code changes required for AWS-managed services. For QE the trace ID emitted by a test request becomes the primary diagnostic tool — when an integration test fails, I pass the trace ID to X-Ray and see the complete failure path in seconds, rather than correlating logs across four Lambda functions manually.

30What is AWS Secrets Manager and how does it affect test automation?Security▾
3-sentence answer
AWS Secrets Manager stores and rotates credentials — database passwords, API keys, third-party service tokens — without hardcoding them in application or test code. In test automation this means test scripts retrieve credentials from Secrets Manager at runtime rather than from environment variables in the CI configuration — if the production password rotates, the test still works because it fetches the current secret rather than a stale copy. I also test the Secrets Manager integration itself — validating that the Lambda correctly retrieves and caches the secret, that rotation events don't break the application (a common production failure when rotation happens mid-execution), and that the IAM role for the Lambda has read access to the correct secret ARN but not to other secrets.

31How do you test AWS WAF (Web Application Firewall) rules?Security▾

38How do you approach QE for a system that processes 100 million financial transactions per month?Strategy▾
3-sentence answer
At 100 million transactions per month the failure modes and their consequences are fundamentally different from a typical application — a 0.1% error rate means 100,000 failed financial transactions per month, and a data integrity bug in the ETL pipeline could corrupt financial records that feed into regulatory reporting. My QE approach for FCS-class systems: zero-tolerance data validation (every transaction output is reconciled against its input — no approximations, no statistical sampling), chaos engineering as a standard practice (what happens to in-flight transactions when a Lambda fails mid-Step-Functions execution?), and formal load testing at 120% of peak monthly volume compressed into a 30-minute window to validate the system's burst capacity before month-end close. The quality standard for a financial transaction system is fundamentally different from a consumer app — and I'd design the quality framework to reflect that difference explicitly.
Liberty Mutual's FCS processes 100M transactions in one run at month-end. The QE framework for this system must be designed around that batch processing model — not continuous transaction flow.
